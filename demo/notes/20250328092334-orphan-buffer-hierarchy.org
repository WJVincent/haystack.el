#+TITLE: Buffer Hierarchy in Haystack
#+DATE: 2025-03-28
# %%% haystack-end-frontmatter %%%
Buffer hierarchy in Haystack refers to the parent-child structure of search and
filter results buffers. A root search (via ~haystack-run-root-search~) creates
a top-level buffer with no parent. Each progressive filter (via =f= in that
buffer) creates a child buffer whose ~haystack--parent-buffer~ points back to
the buffer it was filtered from. This chain can grow several levels deep: a
root search for "emacs," filtered by "org-mode," filtered again by "capture"
produces a three-level chain. The tree view (~haystack-show-tree~) renders
this hierarchy as an indented outline. Navigation commands operate on this
structure: =u= (~haystack-go-up~) moves to the parent buffer, =d=
(~haystack-go-down~) moves to a child, and ~haystack-go-root~ walks the
parent chain all the way to the root. Kill commands also follow the hierarchy:
~haystack-kill-node~ kills just one buffer, ~haystack-kill-subtree~ kills a
buffer and all its descendants, and ~haystack-kill-whole-tree~ walks to the
root and kills the entire tree. ~haystack-kill-orphans~ cleans up buffers
whose parent has been killed but which have no living children of their own.
The hierarchy is entirely determined by search and filter operations — note
content, links between notes, and content similarity play no role in the
buffer tree structure. Multiple independent search sessions produce
disconnected trees in the forest, each with its own root and filter chain.
