title: Tail Recursion
date: 2025-02-14
---

Tail recursion occurs when a recursive call is the last operation in a function
— the result of the recursive call is returned directly without any further
computation. Languages that implement tail call optimization (TCO) convert tail-
recursive functions into loops at the compiler level, eliminating stack frame
growth and making deep recursion safe. Scheme mandates TCO in all conforming
implementations, making proper tail recursion a language feature rather than an
optional compiler optimization. Common Lisp (cl) does not require TCO, but some
implementations (SBCL, CCL) perform it in certain cases; portable CL code should
not rely on it for correctness. Emacs Lisp (emacs-lisp) does not perform TCO, so
deeply recursive elisp functions risk stack overflow on large inputs; the
standard remedy is to rewrite them with `cl-loop` or explicit accumulator
iteration. Clojure addresses the JVM's lack of guaranteed TCO with the `recur`
special form: any function using `recur` in tail position is guaranteed to
compile to a loop rather than a recursive call. The accumulator pattern is the
standard transformation for enabling TCO: add an extra parameter that carries
the accumulated result, and the base case returns the accumulator directly.
Mutual tail recursion — two functions each calling the other in tail position —
requires a trampoline in languages without proper TCO: a loop that repeatedly
calls thunks until a non-function value is returned. In practical elisp code,
tail recursion concerns arise when processing large note corpora or deeply
nested data structures; iterative alternatives are always available via
`dolist`, `cl-loop`, or `while`. Understanding tail recursion helps explain the
design decisions in each Lisp dialect and informs the choice of recursion style
for any production Lisp code.
