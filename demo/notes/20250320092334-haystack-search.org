#+TITLE: Haystack Search
#+DATE: 2025-03-20
# %%% haystack-end-frontmatter %%%
Search is the central operation in Haystack: the ~haystack-run-root-search~
command launches a ripgrep (rg) query against the notes directory and presents matching
notes in a grep-mode results buffer that supports progressive filtering. Ripgrep
(rg) is chosen as the search backend because it is the fastest general-purpose
text search tool available: on a corpus of 10,000 notes, rg completes in well
under 100ms even on modest hardware. Haystack passes the user's query to rg with
appropriate flags — case-insensitive by default — and presents matching lines in
grep-mode format (file:line:content), so compile-goto-error navigates directly
to the match. The frecency engine records search chains and provides a
completing-read interface for replaying past searches, ranking by a score that
combines recency and frequency. Expansion
groups multiply the effective recall of a search: a query for "zettelkasten"
also finds notes containing "pkm," "knowledge-management," and "second-brain" if
those terms are grouped together. The results buffer is a grep-mode buffer, so
standard Emacs navigation works: RET visits the match, n/p moves between lines,
and f filters further within the current results. Search scope prefixes (=>= for
body-only, =<= for frontmatter-only) control which part of each note is searched,
using the frontmatter sentinel as the boundary.
For a second-brain with thousands of zettel, the combination of fast rg search
and progressive filtering makes Haystack feel qualitatively different from simple
grep: it finds the right note, not just notes that match. Search in Haystack is
not merely a lookup operation — it is the primary navigation mechanism,
replacing folder browsing and tag hierarchies with a single fast, expressive
interface.
